# Phase 6.5 Integration Tests Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Create `tests/integration/test-integration.sh` — 9 black-box subprocess tests that prove `bin/shql`'s full CLI pipeline works end-to-end.

**Architecture:** All tests invoke `bin/shql` as a subprocess via a `_shql()` helper that injects `SHELLFRAME_DIR` and `XDG_DATA_HOME`. No module sourcing — every assertion goes through the real CLI binary. Tests use the existing ptyunit assert framework (`assert_eq`, `assert_contains`) exactly as the other integration tests do.

**Tech Stack:** bash, sqlite3, ptyunit assert framework (`tests/ptyunit/assert.sh`), existing fixture `tests/fixtures/demo.sqlite`

Write exactly this content:

```bash
#!/usr/bin/env bash
# tests/integration/test-integration.sh — End-to-end CLI integration tests
# REQUIRES: sqlite3 binary on PATH, SHELLFRAME_DIR set in environment

set -u
TESTS_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
SHQL_ROOT="$(cd "$TESTS_DIR/.." && pwd)"

source "$TESTS_DIR/ptyunit/assert.sh"

if ! command -v sqlite3 >/dev/null 2>&1; then
    printf 'SKIP: sqlite3 not found\n'
    ptyunit_test_summary
    exit 0
fi

# ── Setup ─────────────────────────────────────────────────────────────────────

_data_dir=$(mktemp -d)
_db="$SHQL_ROOT/tests/fixtures/demo.sqlite"

_shql() {
    SHELLFRAME_DIR="$SHELLFRAME_DIR" \
    XDG_DATA_HOME="$_data_dir" \
    bash "$SHQL_ROOT/bin/shql" "$@"
}

# ── Tests will go here ────────────────────────────────────────────────────────

# ── Teardown ──────────────────────────────────────────────────────────────────

rm -rf "$_data_dir"
ptyunit_test_summary
```

```bash
# ── query-out mode ────────────────────────────────────────────────────────────

ptyunit_test_begin "query-out: exit 0 on valid query"
_rc=0; _out=$(_shql "$_db" -q "SELECT id FROM users" 2>/dev/null) || _rc=$?
assert_eq 0 "$_rc"

ptyunit_test_begin "query-out: formatted table output + real data"
assert_contains "$_out" "+"
assert_contains "$_out" "Alice"

ptyunit_test_begin "query-out: --porcelain suppresses box formatting, data present"

assert_contains "$_out" "Alice"
_found=0; [[ "$_out" == *"+"* ]] && _found=1
assert_eq 0 "$_found"
```

Note: `assert_contains` has no negation form — the spec uses `[[ ... ]]` to check absence and asserts the result is 0.

_rc=0; _err=$(_shql "$_db" -q "SELECT * FROM nonexistent_table" 2>&1 >/dev/null) || _rc=$?
assert_eq 1 "$_rc"
assert_contains "$_err" "nonexistent_table"

```

```

If error path tests fail:
- Exit code is 0 when it should be 1 → `bin/shql` is not propagating the error exit code; check the `query-out|pipe` dispatch and the `_rc=$?` path in `bin/shql`
- Stderr is empty → the error message is being swallowed; confirm `_err_tty` routing in `bin/shql` sends to stderr (not only `/dev/tty`) in non-interactive contexts
- Substring not found → the error message format differs from what's expected; run the failing `_shql` call manually and inspect stderr to find the stable substring

**Note on stderr capture:** The `2>&1 >/dev/null` redirect inside `$(...)` works as follows: `2>&1` duplicates the current stdout fd (the `$(...)` capture pipe) into stderr; `>/dev/null` then redirects stdout to `/dev/null`, leaving stderr pointing at the capture pipe. Result: stderr captured, stdout discarded.
