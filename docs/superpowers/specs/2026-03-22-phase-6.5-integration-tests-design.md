# Phase 6.5 — Integration Tests Design

**Date:** 2026-03-22
**Issue:** [shellql#9](https://github.com/fissible/shellql/issues/9)
**Effort:** M (~half day)

---

## Goal

Prove that the full CLI pipeline works end-to-end: argument parsing → dispatch →
adapter → formatter → stdout. The adapter (`test-db.sh`) and parser (`test-cli.sh`)
are already unit-tested. Phase 6.5 adds one black-box subprocess test file that
exercises `bin/shql` directly.

---

## Scope

**In scope:**
- Non-TUI CLI modes: `query-out`, `pipe`, `databases`
- Discovery round-trip: push via `query-out` → assert in `databases` output
- Error paths: missing DB file, bad SQL

**Out of scope:**
- TUI modes (`welcome`, `open`, `table`, `query-tui`) — validated by Phase 5 mock
  screens; PTY/TUI integration tests are future scope
- `sqlite3 not found` error path — already covered by `test-db.sh` unit tests
- Unwritable DB — edge case; adds filesystem setup complexity for low value
- Short-name resolution (`shql demo -q ...`) — resolver not yet implemented;
  filed as a follow-up ticket

## Structure and Setup

### Preamble

```bash
#!/usr/bin/env bash
# tests/integration/test-integration.sh — End-to-end CLI integration tests
# REQUIRES: sqlite3 binary on PATH, SHELLFRAME_DIR set in environment

set -u
TESTS_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
SHQL_ROOT="$(cd "$TESTS_DIR/.." && pwd)"

source "$TESTS_DIR/ptyunit/assert.sh"
```

`SHQL_ROOT` is derived the same way as all existing test files (two `cd` levels up
from the test file's directory). `assert.sh` must be sourced before any assertions.

### Guards

```bash
if ! command -v sqlite3 >/dev/null 2>&1; then
    printf 'SKIP: sqlite3 not found\n'
    ptyunit_test_summary
    exit 0
fi
```

No `SHELLFRAME_DIR` guard. If shellframe is not configured, `bin/shql` will emit
its own error and the tests will fail — which is the correct behaviour for a runner
configuration error.

### Setup

```bash
_data_dir=$(mktemp -d)
_db="$SHQL_ROOT/tests/fixtures/demo.sqlite"

_shql() {
    SHELLFRAME_DIR="$SHELLFRAME_DIR" \
    XDG_DATA_HOME="$_data_dir" \
    bash "$SHQL_ROOT/bin/shql" "$@"
}
```

- `_data_dir` — isolated temp dir; overrides `XDG_DATA_HOME` so `bin/shql`'s
  `src/state.sh` derives `SHQL_DATA_DIR` pointing into the temp dir, not
  `~/.local/share/shellql`. Because `_shql` runs a fresh subprocess each call,
  `SHQL_DATA_DIR` is re-derived from `XDG_DATA_HOME` on every invocation —
  no caching concern.
- `_db` — existing fixture with tables: `users` (Alice, Bob, Carol), `orders`,
  `products`, `categories`.
- `_shql` — wrapper that injects the correct env before every `bin/shql` call.

### Test block structure

Each test is a labelled inline block — no shell functions. Use `ptyunit_test_begin`
to set the name, then run assertions. Example:

```bash
ptyunit_test_begin "query-out: exit 0 on valid query"
_rc=0; _out=$(_shql "$_db" -q "SELECT id FROM users" 2>/dev/null) || _rc=$?
assert_eq 0 "$_rc"
```

All assertions accumulate into a global pass/fail count — `assert_eq` and
`assert_contains` never call `exit`.

### Teardown

```bash
rm -rf "$_data_dir"
ptyunit_test_summary
```

`rm -rf` runs before `ptyunit_test_summary`. `ptyunit_test_summary` uses `return`
not `exit` — cleanup is guaranteed either way, but this order is idiomatic.

## Test Inventory (9 tests)

### query-out mode (3 tests)

1. **Exit 0 on valid query** — `_shql "$_db" -q "SELECT id FROM users"` returns 0

2. **Formatted table output + real data** — same invocation; stdout contains `+`
   (border characters, proves `shql_cli_format_table` is wired) AND contains
   `Alice` (real row from fixture, proves the adapter round-trip works)

3. **--porcelain suppresses formatting** — `_shql "$_db" -q "SELECT name FROM users" --porcelain`;
   stdout does not contain `+`, but does contain `Alice`

### pipe mode (2 tests)

4. **Exit 0 + data present** —
   `printf 'SELECT name FROM users LIMIT 1' | _shql "$_db"` exits 0 and stdout
   contains `Alice`

5. **--porcelain with pipe** — same SQL piped with `--porcelain`; stdout does not
   contain `+`, but contains `Alice`

### databases mode (2 tests — round-trip)

A plain `query-out` call (no `--porcelain`) is the precondition. `bin/shql`
calls `shql_conn_push` after a successful query regardless of output mode, so
the push always fires when exit code is 0. `shql_conn_push` is idempotent —
repeated pushes of the same path upsert rather than append, so tests 6 and 7
may share a `_data_dir` without order-dependence or duplicate-row concerns.

6. **Path appears in output** — after `_shql "$_db" -q "SELECT 1"` exits 0,
   `_shql databases` exits 0 and stdout contains `$_db`

7. **--porcelain variant** — `_shql databases --porcelain` stdout contains `$_db`
   as raw TSV

### Error paths (2 tests)

8. **Missing DB file** — `_shql /tmp/_shql_no_such_db -q "SELECT 1"` returns exit
   code 1 and stderr contains the bare substring `/tmp/_shql_no_such_db` (no
   surrounding quotes; the path is echoed in the error message and is stable
   regardless of exact message wording)

9. **Bad SQL** — `_shql "$_db" -q "SELECT * FROM nonexistent_table"` returns exit
   code 1 and stderr contains `"nonexistent_table"` (sqlite3 error includes the
   table name; stable substring regardless of sqlite3 version)

## Assertion Conventions

Follows `test-db.sh` exactly:

- `assert_eq` for exit codes and exact values
- `assert_contains` for output content
- Stderr captured via `2>&1 >/dev/null` at the call site when needed

**Exit code capture (all modes):**
```bash
_rc=0; _out=$(_shql ... 2>/dev/null) || _rc=$?
assert_eq 0 "$_rc"
```

**Stderr capture (error paths):**
```bash
_rc=0; _err=$(_shql ... 2>&1 >/dev/null) || _rc=$?
assert_eq 1 "$_rc"
assert_contains "$_err" "some_stable_substring"
```

Inside `$(...)`, the subshell's stdout is the capture pipe. `2>&1` duplicates
the current stdout fd (the capture pipe) into stderr's fd. `>/dev/null` then
redirects stdout to `/dev/null` — but stderr already points at the original
capture pipe and is unaffected. Net result: stderr is captured, stdout is
discarded. This is correct.

**Pipe mode** — uses `_rc` idiom to capture exit code:
```bash
_rc=0
_out=$(printf 'SELECT name FROM users LIMIT 1' | _shql "$_db" 2>/dev/null) || _rc=$?
assert_eq 0 "$_rc"
assert_contains "$_out" "Alice"
```
Run inside `$(...)` so stdin is a real pipe. `bin/shql` uses `[ -p /dev/stdin ]`
to detect pipe mode; this check works on Linux and macOS (verified in the existing
`test-cli.sh` pipe tests which use the same pattern).

**databases round-trip:**
```bash
_shql "$_db" -q "SELECT 1" >/dev/null 2>&1       # push to registry
_rc=0; _out=$(_shql databases 2>/dev/null) || _rc=$?
assert_eq 0 "$_rc"
assert_contains "$_out" "$_db"
```

## Dependency Chain Covered

```
bin/shql
  ├── src/cli.sh         (shql_cli_parse → _SHQL_CLI_MODE, _SHQL_CLI_SQL, etc.)
  ├── src/db.sh          (shql_db_query → real sqlite3)
  ├── src/connections.sh (shql_conn_push → shellql.db registry)
  └── src/cli.sh         (shql_cli_format_table → MySQL-style box output)
```

Each test exercises a different path through this graph. The `databases` round-trip
additionally exercises `shql_conn_list` and the registry persistence layer.
